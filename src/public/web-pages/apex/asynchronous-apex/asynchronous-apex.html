Asynchronous Apex

Before understanding asynchronous apex you have to understand the below concepts.

  1. Program
  2. Process
  3. Thread
  4. Transaction

A program becomes a process, a process contains threads, and each thread executes work within a transaction context.

Program

A program is a set of instructions that is written in a particular language for example apex, java, python, etc.

  • It does not do anything, it is just a file that is stored in the database.

Process (Program Execution)

When you execute the program it will become a process.

  • A process has its own memory and resource.
  • A process can execute in multiple threads.

Thread( Execution Unit)

Smallest unit of execution that is part of a process.

  • Each thread can create multiple transaction

Transaction (Execution Protection)

A group of operations that succeed or fail together basically transaction is a logical execution boundary that ensures data consistency and governor limit.

The most important thing is that apex is not a single threaded language but each apex request follows a single-threaded execution model per transaction.

  • Each request is handled by a single thread and executed within one transaction.
  • While parallelism is achieved through multiple independent transactions using asynchronous Apex.

Beyond Understanding

When Account.Phone is updated → all related Contacts.Phone should also be updated.

Synchronous Approach

  1. You will write apex class and apex trigger handler (Program)
  2. When a user updates the phone on account the program execution starts also known as process and Salesforce receives the request.
  3. It will assign a single thread, and execute all automation—Apex triggers, flows, workflows, and contact record updates and its automation run within a single transaction.

Order of execution:

Account automation (Trigger, Flow, VR etc.)

↓

Contact updates (from Apex)

Contact automation (Trigger, Flow, VR  etc.)

↓

Transaction commits OR rolls back

Synchronous approach ensures atomicity, meaning if any operation fails, the entire transaction is rolled back.

Asynchronous Approach

  1. You will write apex class and apex trigger handler (Program)
  2. When a user updates the phone on account the program execution starts also known as process and Salesforce receives the request.
  3. It will assign a single thread, and execute all automation—Apex triggers, flows, workflows related to account —within a single transaction.
  4. But when apex execution reaches to asynchronous apex then the Async job is queued first, then executed later in a new thread & transaction and update on contact will happen and all its automation contact apex trigger flow will run in that transaction.

User updates Account

↓

Thread 1 → Transaction 1

(Account automation + Async job enqueued) →.. →..

↓

(After some time)

↓

Thread 2 → Transaction 2

(Contact update + Contact automation) →.. →..

In the asynchronous approach, the initial execution runs in one transaction, and async processing is executed later in a separate thread and transaction, making it independent.

Asynchronous Apex

Asynchronous Apex is a way to run Apex code in the background, separately from the current execution, using a different thread and transaction.
